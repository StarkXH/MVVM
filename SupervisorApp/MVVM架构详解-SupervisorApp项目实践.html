MVVM架构详解 - SupervisorApp项目实践

🏗️ MVVM架构概述

MVVM (Model-View-ViewModel) 是一种设计模式，特别适合WPF应用程序，它将应用程序分为三个核心层次：

┌─────────────┐    ┌──────────────┐    ┌─────────────┐
│    View     │◄──►│  ViewModel   │◄──►│    Model    │
│   (UI层)    │    │  (逻辑层)    │     │  (数据层)   │
└─────────────┘    └──────────────┘    └─────────────┘

📱 1. View 层 - 用户界面

职责

  • 🎨 纯UI展示：只负责用户界面的显示和用户交互
  • 🚫 无业务逻辑：不包含任何业务逻辑代码
  • 🔗 数据绑定：通过DataBinding与ViewModel交互

🧠 2. ViewModel 层 - 逻辑处理

职责

  • 🔄 数据转换：将Model数据转换为View可用的格式
  • 🎮 命令处理：处理用户操作（按钮点击、输入等）
  • 📊 状态管理：管理UI状态和业务流程
  • 🔔 事件通知：实现INotifyPropertyChanged，通知UI更新

实际代码示例 

// RegisterMapViewModel.cs - 典型的ViewModel
public class RegisterMapViewModel : ViewModelBase
{
    // 🔗 引用Model
    private IDevice _currentDevice;
    
    // 🎮 UI命令
    public RelayCommand ConnectCommand { get; private set; }
    public AsyncRelayCommand RefreshAllCommand { get; private set; }
    
    // 📊 UI状态
    public bool IsConnected { get; set; }
    public string DeviceStatus { get; set; }
    
    // 🔄 业务逻辑
    private async Task ConnectAsync()
    {
        // 调用Model进行设备连接
        await CurrentDevice.InitializeAsync();
        IsConnected = CurrentDevice.ConnectionState == DeviceConnectionState.Ready;
    }
}

💾 3. Model 层 - 数据和业务

职责

  • 📦 数据封装：封装应用程序的核心数据
  • 🔧 业务逻辑：实现核心业务规则和算法
  • 🗄️ 数据持久化：处理数据存储和检索
  • 🌐 外部通信：与硬件、网络、数据库等交互

🔄 4. 三层交互关系

数据流向 

用户操作 → View → ViewModel → Model → 数据处理 → ViewModel → View → UI更新

项目中的实际交互

示例1：设备连接操作 

// 1. View: 用户点击连接按钮
<Button Command="{Binding ConnectCommand}" Content="Connect" />

// 2. ViewModel: 处理命令
public RelayCommand ConnectCommand => new RelayCommand(
    () => _ = ExecuteAsync(ConnectAsync, "Connecting to device..."));

// 3. Model: 执行实际连接
await CurrentDevice.InitializeAsync(); // TestDevice100

// 4. ViewModel: 更新状态
IsConnected = CurrentDevice.ConnectionState == DeviceConnectionState.Ready;

// 5. View: 自动更新UI（通过数据绑定）

示例2：日志显示 

// 1. Model: 产生日志数据
LogService.Instance.LogInfo("Device connected");

// 2. ViewModel: 无需处理（直接绑定）

// 3. View: 自动显示更新
<TextBox Text="{Binding Source={x:Static common:LogService.Instance}, Path=LogText}" />

✅ 6. MVVM的优势

分离关注点

  • 🎨 UI设计师专注View层美化
  • 💻 开发人员专注ViewModel和Model逻辑
  • 🧪 测试人员可以独立测试业务逻辑

可维护性

  • 🔧 修改UI不影响业务逻辑
  • 🔄 修改业务逻辑不影响UI
  • 🎯 每层职责清晰，bug容易定位

可测试性 

// 可以独立测试ViewModel逻辑
[Test]
public void ConnectCommand_ShouldUpdateIsConnected()
{
    var viewModel = new RegisterMapViewModel();
    var mockDevice = new MockDevice();
    viewModel.CurrentDevice = mockDevice;
    
    viewModel.ConnectCommand.Execute(null);
    
    Assert.IsTrue(viewModel.IsConnected);
}

🎯 7. 最佳实践建议

View层最佳实践

  • ✅ 只包含UI声明和简单的事件处理
  • ✅ 使用数据绑定而不是代码操作UI
  • ❌ 避免在View中写业务逻辑

ViewModel层最佳实践

  • ✅ 实现INotifyPropertyChanged接口
  • ✅ 使用RelayCommand处理用户命令
  • ✅ 保持与View的松耦合
  • ❌ 避免直接引用View控件

Model层最佳实践

  • ✅ 封装核心业务逻辑
  • ✅ 提供清晰的接口定义
  • ✅ 处理数据验证和错误
  • ❌ 避免依赖UI框架

这就是MVVM架构在你的SupervisorApp项目中的实际应用！每一层都有明确的职责分工，通过数据绑定和命令模式实现松耦合的架构设计。
